# Convention de documentation du code

Objectif: rendre le portage moderne lisible dans la duree, en gardant un lien clair entre le code TypeScript, le comportement ISO vise et les preuves ASM/extraction.

Cette convention s'applique a tout le code source du projet, notamment `src/`, `tools/` et les fichiers de generation maintenus a la main.

## Principes

- [ ] Chaque fichier doit commencer par un commentaire d'entete.
- [ ] Chaque entite significative doit etre documentee.
- [ ] La documentation doit expliquer l'intention, pas paraphraser le code.
- [ ] Les comportements ISO, ASM ou extraction doivent etre signales explicitement.
- [ ] Les choix modernes doivent etre distingues des preuves originales.
- [ ] Les commentaires doivent rester courts, precis et maintenables.
- [ ] Les fichiers generes peuvent avoir une entete courte indiquant le generateur.

## Entete obligatoire de fichier

Chaque fichier TypeScript doit commencer par un bloc `/** ... */`.

Format recommande:

```ts
/**
 * Role: Decrit la responsabilite principale du fichier.
 * Scope: Precise ce que ce fichier a le droit de faire.
 * ISO: Indique si le fichier porte une regle prouvee TO8/ASM/extraction.
 * Notes: Mentionne les contraintes ou dettes connues si necessaire.
 */
```

Exemple:

```ts
/**
 * Role: Charge les niveaux JSON modernes et les convertit en LevelDefinition runtime.
 * Scope: Valide le format moderne, cree la grille initiale et les entites de niveau.
 * ISO: Les tile ids runtime produits correspondent aux identifiants prouves dans runtime-tiles.ts.
 * Notes: Les JSON modernes ne doivent pas contenir d'adresses ASM.
 */
```

## Entete des fichiers generes

Les fichiers generes doivent garder une entete minimale:

```ts
/**
 * Generated by tools/extract-example.mjs.
 * Do not edit by hand.
 * Source: docs/extraction/example-metadata.json.
 */
```

Si le fichier est regenere automatiquement, ne pas ajouter de documentation manuelle qui serait perdue.

## Documentation des constantes

Toute constante top-level doit avoir un commentaire.

Format:

```ts
/** Description courte, avec provenance si utile. */
const EXAMPLE_CONSTANT = 123;
```

Pour les constantes ASM/provenance:

```ts
/** Seuil camera droit observe dans le comportement original TO8. */
const CAMERA_RIGHT_MARGIN = 0x0f;
```

Pour les constantes modernes:

```ts
/** Duree moderne d'interpolation visuelle; ne change pas la logique discrete case par case. */
const PLAYER_GRID_MOVE_DURATION = 0.21;
```

## Documentation des types et interfaces

Tout type ou interface doit documenter son role.

Les champs non triviaux doivent aussi etre commentes.

Exemple:

```ts
/** Representation moderne editable d'un niveau avant conversion runtime. */
export interface ModernLevelJson {
  /** Identifiant stable du niveau moderne, sans adresse ASM. */
  readonly id: string;

  /** Largeur logique de la grille en cellules de 16 pixels. */
  readonly width: number;
}
```

## Documentation des fonctions

Toute fonction top-level et toute methode de classe doit avoir un commentaire.

Format minimal:

```ts
/** Explique ce que fait la fonction et pourquoi elle existe. */
function example(): void {
  // ...
}
```

Format recommande pour logique non triviale:

```ts
/**
 * Determine la prochaine cellule cible d'un objet physique.
 * Respecte la chute verticale directe puis la bascule laterale sous contrainte de support.
 */
function resolveFallingObjectTarget(): Target | null {
  // ...
}
```

Quand la fonction applique une regle ISO:

```ts
/**
 * Applique la priorite de deplacement des monstres telle que reproduite depuis le runtime.
 * Les directions gardent le codage historique 1..4.
 */
function advanceSingleMonsterRuntime(): void {
  // ...
}
```

## Documentation des variables de classe

Tout champ de classe doit etre documente.

Exemple:

```ts
/** Etat courant de la camera logique, en coordonnees de grille. */
private readonly viewport: ViewportState;

/** Mouvement camera en cours, utilise uniquement pour l'interpolation visuelle. */
private cameraMove: CameraMoveState | null = null;
```

## Documentation des variables locales

Les variables locales simples n'ont pas besoin de commentaire systematique si leur nom est explicite.

Une variable locale doit etre commentee si:

- elle porte une valeur issue de l'ASM;
- elle contourne un cas particulier;
- elle encode une convention de grille/runtime;
- elle explique une exception ISO.

Exemple:

```ts
// `0x80` est une trace runtime de monstre: elle se rend comme du vide et doit etre traversable.
const isMonsterTrail = tileId === MONSTER_RUNTIME_TRAIL_TILE_ID;
```

## Documentation des exports

Tout export public doit documenter:

- son role;
- qui devrait l'utiliser;
- si l'export est une facade de compatibilite;
- si l'export expose une convention runtime.

Exemple:

```ts
/** Facade stable pour creer l'etat initial d'un niveau gameplay. */
export function createGameLevelState(levelNumber = 1): GameState {
  // ...
}
```

## Documentation des imports particuliers

Les imports n'ont pas besoin de commentaire.

Exception: si un import semble surprenant ou risqué, ajouter une note pres de l'usage, pas sur la ligne d'import.

Exemple:

```ts
// Les metadata de sprites restent la source runtime des cycles d'animation extraits.
const playerFrames = extractFrameIdsFromMetadata(...);
```

## Documentation des exceptions et dettes

Les exceptions doivent etre marquees explicitement.

Format recommande:

```ts
// NOTE: ...
// TODO: ...
// FIXME: ...
```

Definitions:

- `NOTE`: explique une contrainte volontaire.
- `TODO`: travail prevu mais non bloquant.
- `FIXME`: comportement connu comme faux ou fragile.

Exemple:

```ts
// NOTE: La mutation reste immediate pour conserver le comportement discret actuel.
```

## Documentation liee a l'ASM/provenance

Quand une regle vient du code original, le commentaire doit mentionner la source si elle est connue.

Exemple:

```ts
/**
 * Tile runtime vide.
 * Provenance: routines de chute/collision `CB3B`, `CBA7`, `BD03`.
 */
export const RUNTIME_EMPTY_TILE_ID = RUNTIME_TILE.empty;
```

Si la source n'est pas encore prouvee:

```ts
/**
 * Choix moderne temporaire en attente de verification ASM.
 * Ne pas traiter comme une preuve ISO.
 */
```

## Documentation des fichiers de plan et markdown

Les fichiers `.md` de plan doivent utiliser:

- des phases;
- des items cochables `[ ]` / `[x]`;
- une section `Notes` quand une decision importante est prise;
- une distinction claire entre fait, suppose et a verifier.

## Niveau de detail attendu

La documentation doit etre complete mais pas bruyante.

Bon commentaire:

```ts
/** Convertit les cellules modernes en tile ids runtime TO8. */
```

Mauvais commentaire:

```ts
/** Incremente i de 1. */
i += 1;
```

## Ordre recommande dans un fichier

1. Entete de fichier.
2. Imports.
3. Types/interfaces exportes.
4. Constantes exportees.
5. Constantes internes.
6. Classes/fonctions exportees.
7. Fonctions internes.
8. Helpers purs.

## Definition de fini pour un fichier

Un fichier est considere conforme si:

- [ ] il possede une entete;
- [ ] chaque type/interface est documente;
- [ ] chaque constante top-level est documentee;
- [ ] chaque fonction/methode est documentee;
- [ ] chaque champ de classe est documente;
- [ ] les variables locales complexes sont commentees;
- [ ] les choix ISO/provenance sont identifies;
- [ ] les dettes ou exceptions sont marquees `NOTE`, `TODO` ou `FIXME`;
- [ ] les commentaires expliquent l'intention plutot que la syntaxe.

## Strategie d'application progressive

Ne pas tenter de documenter tout le repo en une seule passe si cela ralentit le portage.

Ordre conseille:

1. Documenter les fichiers runtime critiques: `runtime-tiles`, `runtime-grid`, `level-loader`.
2. Documenter les systems gameplay.
3. Documenter `GameplayScene`.
4. Documenter les renderers.
5. Documenter les tools d'extraction modifies.
6. Documenter les fichiers generes uniquement via leurs generateurs.
